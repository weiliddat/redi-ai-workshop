Here’s a simple breakdown of what a callback function is and why it is used specifically in WebSocket programming.

### 1. What is a Callback Function?

In programming, a **callback function** is simply a function that you pass as an argument (data) to another function, so that the second function can "call it back" later.

Think of it like ordering food at a restaurant:

1.  You give the cashier your phone number (you pass the function).
2.  You go sit down and do other things (the code continues running).
3.  When the food is ready, the kitchen calls your number (the callback is executed).

**In code:**

```javascript
function orderFood(phoneNumber) {
    console.log("Cooking food...");
    // Later, when food is ready:
    phoneNumber(); // This calls the callback
}

function myPhoneRing() {
    console.log("Hello, my food is ready!");
}

// You pass the function 'myPhoneRing' as an argument
orderFood(myPhoneRing); 
```

### 2. Why pass a function to `ws.onmessage`?

When working with WebSockets (like `ws.onmessage`), you aren't writing a linear script (top to bottom). You are building a **reactive system**.

Here is why you pass a function (the callback) instead of writing the code directly:

#### A. Asynchronous Nature (The "When")
WebSockets are **event-driven**. You don't know *when* a message will arrive from the server. It could be in 1 millisecond or 10 minutes from now.

*   **If you wrote code directly:** The script would try to read a message immediately, find nothing, and crash or stop.
*   **With a callback:** You are saying: *"Hey WebSocket, here is a bundle of instructions (the function). Keep this safe, and only run it **when** a message actually arrives."*

#### B. Non-Blocking (Doing other things)
JavaScript is single-threaded. If your code stopped and waited (blocked) for a message to arrive, your entire application would freeze—buttons wouldn't click, animations would stop, etc.

By passing a callback, you tell the system: *"Don't wait here. Go do other stuff (render the UI, handle clicks), and run this function later when the data shows up."*

#### C. Reusability and Scope
When you pass a function, that function retains access to the variables available at the moment it was created (this is called a **closure**).

**Example:**
```javascript
const userId = "user_123";

ws.onmessage = function(event) {
    // This function remembers that 'userId' exists 
    // even though it runs seconds or minutes later.
    console.log(`Message for ${userId}:`, event.data);
};
```

If you didn't use a callback, you would have to rely on global variables, which makes code messy and harder to debug.

### 3. Visual Comparison

Imagine you are listening for a doorbell.

**The "Blocking" Way (Bad - like writing code directly):**
You stand at the door staring at the peephole. You cannot eat, sleep, or check your phone until someone rings the bell. You freeze your entire life waiting.

**The "Callback" Way (Good - like `ws.onmessage`):**
You sit on the couch watching TV. You tell your roommate: *"If the doorbell rings, please tell me."*
You continue watching TV (your code runs other logic). When the bell rings, your roommate interrupts you with the news (the callback executes).

### 4. Alternative: Promises and Async/Await

While callbacks are common in older WebSocket libraries, modern JavaScript often uses **Promises** or **Async/Await**. However, even those are just syntactic sugar around the same concept: **waiting for something to happen in the future without blocking the main thread.**

For the native WebSocket API in browsers, `onmessage` is still a callback property.

### Summary

You pass a function to `ws.onmessage` because:
1.  **Timing:** You don't know when the message will arrive.
2.  **Efficiency:** Your program shouldn't freeze while waiting.
3.  **Instruction:** You are packaging a set of instructions to execute later, rather than trying to execute them immediately.
