Here is the breakdown of what a callback function is and why it is essential in asynchronous programming like WebSockets.

### 1. What is a Callback Function?

A **callback function** is simply a function that is passed into another function as an argument and is **executed later**.

Think of it like this: You are at a restaurant. You place an order (start a function). You **wait** at the table. When your food is ready, the waiter rings the bell to call you back. You **execute** (eat) the food only when the waiter calls you.

In programming terms:
*   The "Restaurant" is the system doing the work (like the WebSocket connection).
*   The "Order" is the data or task you want to happen.
*   The "Waiter" is the event listener or the callback function.
*   The "Food" is the result.

### 2. Why pass a function to `ws.onmessage`?

If you try to write the code directly inside the WebSocket setup, your program would **freeze** (block) until the data arrives. Because network data doesn't arrive instantly, your website would feel like it has crashed.

Here is the comparison between the two approaches:

#### The "Wrong" Way (Blocking Code)
If you try to get data immediately, your script has to wait forever if no message arrives. It cannot do anything else in the meantime.

```javascript
// BAD: The script blocks (freezes) here waiting for data.
const message = ws.receive(); 

if (message) {
    console.log("I received:", message);
} else {
    console.log("Still waiting...");
    // While waiting here, the user cannot click buttons or scroll the page.
}
```

#### The "Right" Way (Callback/Listener)
By passing a function to `ws.onmessage`, you are telling the WebSocket library: **"Don't give me the data yet. Just give me the *function* to run, and run that function specifically when the data comes in."**

Your script is free to do other things while waiting.

```javascript
// GOOD: The script does not wait. It keeps running.
// This function is stored up in the memory as a "pending action."
ws.onmessage = (message) => {
    console.log("I received:", message);
    // Only when the message arrives does this block execute.
};
```

### Summary of Benefits

1.  **Non-Blocking:** Your application doesn't freeze while waiting for the network. The user can still click buttons or scroll while the connection is open.
2.  **Decoupling:** The code handling the connection (the WebSocket library) doesn't need to know **what** you want to do with the data. It just knows **who** to call when it arrives. You are responsible for writing the logic inside the callback.
3.  **Event-Driven:** The code only runs when it is actually needed (on-demand), rather than running in the middle of other processes.
