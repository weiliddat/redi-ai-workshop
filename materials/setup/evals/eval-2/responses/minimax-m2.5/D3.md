The **`Uncaught SyntaxError: Unexpected token`** is thrown when the JavaScript parser tries to read a piece of code (or a string that is being turned into code) and encounters something it does not understand.  

When the error appears **exactly when a message arrives**, it almost always means that the code that processes the incoming data is trying to treat the message as JavaScript (or JSON) but the message is either:

1. **Not valid JSON** – e.g., it contains extra characters, trailing commas, line‑breaks, or is empty.
2. **Not a valid JavaScript expression** – e.g., you (or a library) are using `eval()`, `new Function()`, or a dynamic `<script>` tag to run the data as code.
3. **Mangled by the transport** – e.g., the server sends a gzip‑compressed response but the client tries to parse it as plain text, or the response is truncated.

Below is a step‑by‑step guide to pinpoint the problem and fix it.

---

## 1. Look at the raw message *before* any parsing

```js
const ws = new WebSocket('wss://your-server.com/socket');

ws.onmessage = event => {
  // 👇 Log exactly what the network layer gives you
  console.log('Raw message:', event.data);
};
```

- Open the **Network** tab of the browser’s devtools, filter by **WS** (WebSocket) or **Fetch/XHR**, and inspect the frame that arrived. Does it look like proper JSON (`{"foo":"bar"}`) or something else?
- Is the message *empty* (`""`) or only whitespace?

If you see anything like `undefined`, `null`, or a truncated fragment, that’s a clue.

---

## 2. Validate the JSON (if you think it *should* be JSON)

```js
ws.onmessage = event => {
  const raw = event.data;
  try {
    const data = JSON.parse(raw); // <-- throws if raw is not valid JSON
    console.log('Parsed data:', data);
    // …proceed with your app logic
  } catch (e) {
    console.error('Invalid JSON:', raw, e);
  }
};
```

*Common culprits that break `JSON.parse`*  

| Symptom | Likely cause |
|--------|---------------|
| `Unexpected token` **at the very beginning** (`Unexpected token {`) | The string starts with a BOM (`\uFEFF`) or an invisible control character. |
| `Unexpected token` **near the end** (`Unexpected token }`) | Trailing comma (`[1,2,]`), a stray `,`, or a missing closing brace. |
| `Unexpected token u` (`Unexpected token u in JSON at position 0`) | The string is literally `"undefined"` (or the JavaScript primitive `undefined`). |
| Empty string / whitespace only | The server sent an empty payload (e.g., a heartbeat). |

If you confirm that the server **does** send JSON, double‑check the **Content‑Type** header:

```
Content-Type: application/json
```

If the header is missing or wrong, the browser may treat the payload as plain text, and later attempts to parse it as JSON can fail.

---

## 3. Check whether you’re *evaluating* the data

If you (or a library you use) are doing something like:

```js
// ❌ Dangerous – executes the message as JavaScript
eval(event.data);

// or
new Function(event.data)();
```

…then the message **must be a valid JavaScript expression**, not just JSON. Most servers do *not* send executable JS, so you’ll almost always get a syntax error.

**Fix** – Stop using `eval`/`new Function` for data that isn’t meant to be code. If you truly need to run a piece of code sent from the server, make sure the payload is:

- A string that is a complete, syntactically correct JavaScript program (e.g., `({foo:1})` or `function(){…}`), and
- Ideally, delivered over a protected channel (e.g., signed, version‑controlled) because `eval` is a huge security risk.

In the majority of cases, the correct approach is to treat the incoming payload as **data**, not as code.

---

## 4. Server‑Sent Events (SSE) or other streaming APIs

If you’re using `EventSource`:

```js
const es = new EventSource('/stream');
es.onmessage = event => {
  console.log('Raw SSE payload:', event.data);
  // … parse as JSON, CSV, etc.
};
```

The same rules apply: the data must be in the format you expect, and you must not try to execute it unless you deliberately intend to run JavaScript.

---

## 5. Common “gotchas” that produce the exact error

| Situation | Why it fails | How to fix |
|-----------|--------------|------------|
| **Empty message (heartbeat)** | `JSON.parse("")` → Unexpected token | Guard against empty strings: `if (!raw.trim()) return;` |
| **Gzipped response** | Browser decompresses automatically, but you may be reading the raw bytes if you use `fetch` with `response.text()` incorrectly | Use `fetch(..., { headers: { 'Accept-Encoding': 'identity' } })` or handle the decompression yourself. |
| **Server returns HTML** (e.g., a 404 page) | The HTML starts with `<!DOCTYPE html>` → Unexpected token when parsed as JSON | Check the HTTP status code; handle non‑200 responses. |
| **Using a library that auto‑parses** (e.g., `socket.io` with default JSON) | The library tries `JSON.parse` on each frame, but some frames are control messages (e.g., `"2::"`) | Use the library’s built‑in parsing or configure it to skip certain message types. |
| **Binary data** (ArrayBuffer / Blob) | `JSON.parse` on a binary buffer yields a SyntaxError | Convert binary to string *iff* it’s meant to be JSON (e.g., `new TextDecoder().decode(buf)`). |

---

## 6. Quick diagnostic checklist

1. **Console log** the raw `event.data` (or `message.data` for SSE) *before* any parsing.
2. **Network tab** – look at the frame/response body. Does it look like what you expect?
3. **Verify Content‑Type** – it should be `application/json` for JSON payloads.
4. **Wrap parsing in try/catch** – at least you’ll see *what* string caused the error.
5. **If you’re using `eval` / `new Function`** – stop it unless you absolutely must run server‑provided code.
6. **If the message is empty** – add a guard: `if (!raw || !raw.trim()) return;`.

---

## 7. Minimal “safe” example

```js
const ws = new WebSocket('wss://example.com/socket');

ws.onmessage = event => {
  const raw = event.data;

  // Guard against empty / whitespace‑only messages
  if (!raw || !raw.trim()) {
    console.warn('Received empty message – ignored.');
    return;
  }

  try {
    const payload = JSON.parse(raw);
    // ✅ Successfully parsed – now use payload
    console.log('✅ Payload:', payload);
  } catch (e) {
    // The raw string is not JSON – maybe it’s a control message?
    console.error('❌ Not JSON:', raw, e);
    // If you expect non‑JSON control frames, handle them here.
  }
};
```

### TL;DR

The error is a JavaScript parsing failure. When a message arrives, something is trying to turn that message into code (or JSON) and the string is malformed.  

- **First step:** log the raw message (`console.log(event.data)`).  
- **Second step:** verify it is valid JSON (or a valid JS snippet, if you really intend to evaluate it).  
- **Third step:** add a `try/catch` around the parsing/evaluation and guard against empty messages.  

Once you can see exactly what the server is sending, you’ll be able to decide whether the problem is a simple formatting issue, a missing `Content‑Type`, or a misunderstanding about what the message is supposed to contain.
