### What is a Callback Function?
A **callback function** is a function passed as an argument to another function, which is then executed ("called back") later when a specific event occurs or an asynchronous task completes. Callbacks are fundamental to JavaScript's event-driven and asynchronous programming models.

**Key characteristics**:
1. **Inversion of Control**: You delegate control to another function (e.g., a library or framework) to invoke your callback when ready.
2. **Asynchronous Handling**: Enables non-blocking operations (e.g., I/O, timers, network requests).
3. **Event-Driven**: Executes in response to events (e.g., user clicks, WebSocket messages).

---

### Why Use a Callback for `ws.onmessage`?
When working with WebSockets (e.g., `ws.onmessage`), you pass a callback function instead of writing code directly for critical reasons:

#### 1. **Asynchronous Nature of WebSockets**
   - **Problem**: WebSocket messages arrive unpredictably. Your code can't predict *when* (or if) a message will arrive.
   - **Solution**: A callback defers execution until a message is received. Without it, you'd need to poll constantly (inefficient) or block the thread (impossible in JavaScript's single-threaded model).

#### 2. **Event-Driven Architecture**
   - **How it works**: The WebSocket API is designed around events (`onopen`, `onclose`, `onerror`, `onmessage`). When a message arrives, the browser/runtime triggers the `onmessage` callback.
   - **Example**:
     ```javascript
     const ws = new WebSocket("wss://example.com");
     
     // Delegate message handling to a callback
     ws.onmessage = (event) => {
       // This runs ONLY when a message arrives
       console.log("Received:", event.data);
     };
     ```
   - **Alternative (inefficient)**:
     ```javascript
     // Not feasible: Would block the thread indefinitely
     while (true) {
       if (messageReceived) { /* ... */ } // Impossible to implement
     }
     ```

#### 3. **Non-Blocking Performance**
   - JavaScript runs in a single thread. If you write blocking code (e.g., `while(!message)`), the entire UI/thread freezes until a message arrives.
   - Callbacks let the runtime manage timing, keeping the app responsive.

#### 4. **Separation of Concerns**
   - **Setup Code**: Establishes the WebSocket connection.
   - **Callback**: Handles incoming messages.  
     This separation improves readability and maintainability.

#### 5. **Reusability and Flexibility**
   - Callbacks can be named functions or lambdas, reused across connections, or parameterized for different behaviors.
   - Example:
     ```javascript
     function handleMessage(event) {
       // Custom logic (e.g., update UI, process data)
     }

     ws1.onmessage = handleMessage;
     ws2.onmessage = handleMessage;
     ```

### Real-World Analogy: Restaurant Order System
- **Without a callback**:  
  You stand at the kitchen door, blocking all other tasks, waiting indefinitely for your order (inefficient and unrealistic).  
  **→ JavaScript can't do this!**

- **With a callback**:  
  You give your order number (callback) to the staff and return to your table. When your order is ready, they call your number (execute the callback).  
  **→ Efficient and non-blocking!**

### When to Write Code "Directly"?
You **can** write code directly in **synchronous** scenarios (e.g., simple loops, calculations). For example:
```javascript
// Synchronous: Immediate execution
const squared = [1, 2, 3].map(x => x * x); 
console.log(squared); // Directly available
```

But for **asynchronous events** (like WebSockets, timers, HTTP requests), callbacks (or modern alternatives like Promises/async-await) are **essential**.

---

### Modern Alternatives to Callbacks
While `ws.onmessage` uses callbacks, newer patterns simplify async code:
1. **Promises**: Better for single async operations (e.g., `fetch`).
2. **Async/Await**: Syntactic sugar for Promises (not directly applicable to event handlers like `onmessage`).
3. **Event Streams (e.g., RxJS)**: Advanced pattern for complex event streams.

For WebSocket messages, callbacks remain the standard approach due to the continuous event-driven nature.

| **Approach**      | **Use Case**              | **Example**                          |
|-------------------|---------------------------|--------------------------------------|
| **Callback**      | Events (e.g., WebSockets) | `ws.onmessage = (event) => { ... }` |
| **Direct Code**   | Synchronous operations    | `console.log("Hello")`              |

Use callbacks for asynchronous events like `ws.onmessage` to leverage JavaScript's non-blocking strengths. For more complex flows, consider Promises/async-await or libraries like RxJS.
