## What a *callback* is

A **callback** is simply a function that you **give to another piece of code** so that *it* can call (i.e., “call back”) your function later, when something specific happens.

- You **pass** the function **by reference** (its name or an arrow function), you do **not** invoke it immediately (`myFunc()` would run it right now).
- The receiver decides **when** to invoke the callback – after a timer, after a network response, after a user clicks, after a WebSocket receives a message, etc.

```js
// ❌ wrong – you’d be calling the function right now and assigning its return value
ws.onmessage = handleMessage();  

// ✅ correct – you hand the function object; WS will call it later
ws.onmessage = handleMessage;  

// ✅ also fine – an anonymous (arrow) function
ws.onmessage = (event) => {
  console.log('Received:', event.data);
};
```

## Why we pass a function to `ws.onmessage`

### 1. The message **arrives later, at any time**
WebSocket communication is asynchronous.  
When you write:

```js
ws.onmessage = (event) => { … };
```

you’re telling the browser: *“When a new chunk of data shows up, run this code.”* The browser will invoke your function **automatically**, passing the `event` object that contains the data.

If you tried to write the code directly in the main flow:

```js
// THIS DOESN'T WORK – there’s no data yet
const data = ws.receive(); // <-- syntax is made‑up, but you get the idea
```

you’d be trying to read a value that doesn’t exist yet. The program would have to **block** and wait (busy‑wait) for a message, which wastes CPU and makes the UI freeze.

### 2. **Event‑driven** architecture
JavaScript (in the browser or Node) uses an *event loop*. The runtime keeps listening for things (clicks, network data, timers). When an event occurs, it *queues* the associated callback to be executed on the main thread. This lets many things happen “simultaneously” without you having to write explicit loops or polling.

```js
// The rest of your program can keep running while WS listens in the background.
console.log('Script started');
ws.onmessage = e => console.log('Got', e.data);
console.log('Script finished');   // runs immediately, not after a message arrives
```

### 3. **Decoupling** and reusability
By passing a function you keep the *who‑calls‑whom* relationship clear:

- The WebSocket code doesn’t need to know *what* you want to do with a message.
- You can swap the handler at runtime, reuse the same handler for multiple sockets, or have many listeners on the same socket.

```js
function logMessage(e) { console.log('LOG:', e.data); }
function processOrder(e) { orders.update(e.data); }

// Same socket, two different callbacks
ws.onmessage = logMessage;   // later you could reassign: ws.onmessage = processOrder;
```

## Analogy with other callback patterns you’ve likely seen

| Situation | What runs later | Callback example |
|-----------|-----------------|------------------|
| `array.forEach(item => …)` | for each element | `arr.forEach(x => console.log(x))` |
| `setTimeout(fn, 1000)` | after 1 second | `setTimeout(() => alert('Hi'), 1000)` |
| `fs.readFile('file','utf8', fn)` | when file is read | `fs.readFile('f','utf8', (err,data)=>{…})` |
| `button.addEventListener('click', fn)` | on click | `btn.addEventListener('click', e => …)` |

All of them follow the same principle: **you hand a function to some system, and that system invokes it when the appropriate moment arrives**.

## Summary

- **Callback** = a function you hand to another piece of code so it can be called later.
- **`ws.onmessage`** expects a callback because the WebSocket receives data **anytime** after the connection is open. The browser invokes your callback **once for each incoming message**.
- Writing the code directly would require you to wait (block) for a message, which is inefficient and impractical in an event‑driven environment.
- Using a callback keeps your code **non‑blocking**, **responsive**, and **modular**.

So when you do:

```js
ws.onmessage = event => {
  // your logic here
};
```

you’re saying: “When a message arrives, run this logic.” That’s exactly what a callback is for.
