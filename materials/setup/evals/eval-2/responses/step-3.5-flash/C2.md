A **callback function** is a function passed as an argument to another function (or assigned to an event handler) to be executed later when a specific event/condition occurs. It's essentially "calling back" to code you defined earlier.

## Why Use a Callback for `ws.onmessage`?

### 1. **Asynchronous Nature**
WebSocket communication is **non-blocking**. Messages arrive at unpredictable times. A callback lets you:
```javascript
// Setup happens NOW
const ws = new WebSocket('ws://example.com');

// This code runs immediately after setup
console.log('WebSocket connection initiated');

// But the message handler runs LATER when data arrives
ws.onmessage = function(event) {
  console.log('Message received (executed later):', event.data);
};
```

### 2. **Event-Driven Architecture**
`onmessage` is an **event handler**. It follows the pattern:
```
When [event] occurs → Execute [callback function]
```
Without callbacks, you'd need to constantly poll/check for messages (inefficient).

### 3. **Code Organization & Reusability**
```javascript
// Define once, reuse
function handleMessage(event) {
  const data = JSON.parse(event.data);
  updateUI(data);
  logToDatabase(data);
}

// Assign to multiple WebSockets if needed
ws1.onmessage = handleMessage;
ws2.onmessage = handleMessage;

// Or switch handlers dynamically
ws.onmessage = handleCriticalMessages;
// Later...
ws.onmessage = handleNormalMessages;
```

### 4. **Separation of Concerns**
```javascript
// Connection logic (separate from message logic)
function connectWebSocket(url) {
  const ws = new WebSocket(url);
  ws.onopen = () => console.log('Connected');
  ws.onerror = (err) => console.error('Error:', err);
  return ws;
}

// Message handling (separate concern)
function setupMessageHandler(ws) {
  ws.onmessage = (event) => {
    // Only message-related code here
    processMessage(event.data);
  };
}

// Cleaner main flow
const ws = connectWebSocket('ws://api.example.com');
setupMessageHandler(ws);
```

### What If You "Wrote Code Directly"?
You might try:
```javascript
// This DOESN'T WORK as expected:
ws.onmessage = console.log('Message will be here'); // ❌ Runs IMMEDIATELY

// Or worse:
ws.onmessage = console.log(event.data); // ❌ No event variable exists yet
```

The callback **delays execution** until the event actually fires, receiving the `event` object as a parameter at that future time.

## Visual Timeline
```
Time ──────────────────────────────────────────►
      │ Setup           │ Message arrives     │
      │ ws.onmessage =  │ [event triggers]    │
      │ function {...}  │ ↓ callback runs     │
      │                 │ console.log(data)   │
```

## Real-World Analogy
Think of `ws.onmessage = callback` like:
- **Setting up a mail forwarding service** (callback function)
- **Not** writing the letter right now (which would execute immediately)
- The service (WebSocket) will **call you back** (invoke callback) when mail (message) arrives

## Key Takeaway
You pass a function to `ws.onmessage` because:
1. **Messages arrive asynchronously** (unknown future time)
2. **You need to specify WHAT to do** when they arrive
3. **The event object (`event`)** is only available WHEN the message arrives
4. **It follows JavaScript's event-driven pattern** used throughout browsers/Node.js

This pattern appears everywhere in JS: `addEventListener()`, `setTimeout(callback)`, `array.map(callback)`, etc.
