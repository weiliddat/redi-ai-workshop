A **callback function** in JavaScript is a function that is passed as an argument to another function and is executed later in response to some event or after an operation completes. This is a core part of JavaScript's event-driven and asynchronous programming model.

### Why Pass a Function Instead of Writing Code Directly?

When working with APIs like WebSocket, which rely on asynchronous communication, you pass a callback function (instead of writing logic directly) to:

1. **React to Events Dynamically**  
   With `ws.onmessage`, the callback only runs when a message is actually received. This allows your program to handle events *as* they occur, rather than requiring a blocking or polling loop.

2. **Separate Concerns and Promote Reusability**  
   The WebSocket connection object (`ws`) stays generic. By passing in your logic as a callback, you can reuse the same WebSocket code for different behaviors without modifying the connection setup.

3. **Maintain Non-Blocking Behavior**  
   WebSockets operate asynchronously. You don't want your app to freeze while waiting for messages. Callbacks let the main thread continue running while still responding to incoming data when needed.

### Practical Example

```javascript
// Setting up the WebSocket
const socket = new WebSocket('wss://example.com');

// Passing a callback function to handle incoming messages
socket.onmessage = function(event) {
  console.log('Received:', event.data);
  // Process and respond to the message
};

// Or using arrow function syntax (modern)
socket.onmessage = (event) => {
  console.log('Received:', event.data);
};
```

When a message arrives at the server, the WebSocket client triggers the `onmessage` callback with that data.

### Real-World Comparison: Using a Callback vs. No Callback

```javascript
// ❌ Without a callback: Blocking and unresponsive
while (socket.open) {
  if (socket.messageAvailable()) {
    console.log(socket.getMessage());
  }
} // Blocks the main thread

// ✅ With a callback: Asynchronous and efficient
socket.onmessage = (event) => {
  console.log(event.data); // Fires naturally when message arrives
}; // Non-blocking and scalable
```

### Why It Matters with WebSockets

WebSockets are built for real-time, bidirectional communication. You can't predict when the server will send data. Callbacks let you handle incoming messages cleanly:

```javascript
const socket = new WebSocket('wss://example.com');

socket.onmessage = (event) => {
  if (event.data === 'ping') {
    handlePing();
  } else if (event.data.startsWith('user:')) {
    handleNewUser(event.data);
  }
};
```

Without passing a function, you would need to manually poll or write event loops, which would waste resources and hinder responsiveness.

### Key Takeaways

- A callback function is executed when a certain event occurs.
- With `ws.onmessage`, you're passing in a function to handle incoming messages.
- This design is cleaner, non-blocking, and scales well.
- JavaScript and WebSockets rely on this pattern for efficient event handling.

Callbacks give your code flexibility and responsiveness without tying it into rigid control flows — a key strength of JavaScript's event-driven model.
