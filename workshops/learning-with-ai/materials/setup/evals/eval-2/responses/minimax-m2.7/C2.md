**Short definition**

A *callback* is simply a **function that you hand to another piece of code so that code can call it later, when a particular thing happens**.  
In JavaScript functions are *first‑class values*: you can store them in variables, pass them as arguments, return them, etc. That makes callbacks a natural way to say “run this logic **when the WebSocket receives a message**”.

---

## 1.  What does a callback look like?

```js
// named function
function handleMessage(event) {
  console.log('Got:', event.data);
}

// pass the *reference* to the function (no parentheses)
ws.onmessage = handleMessage;        // ✅

// anonymous / arrow function (also a reference)
ws.onmessage = function(event) {    // ✅
  console.log('Got:', event.data);
};

// arrow shorthand
ws.onmessage = (event) => console.log('Got:', event.data); // ✅
```

If you wrote `ws.onmessage = handleMessage()` the function would **run immediately** and its return value (probably `undefined`) would be stored in `onmessage` – which is almost never what you want.

---

## 2.  Why can’t you just write the code directly?

```js
// ❌ What most newcomers try first
ws.onmessage = console.log(event.data);
```

When that line executes, `event` **doesn’t exist yet**. The WebSocket has not received any data, so `event.data` is `undefined` and the code runs *right now* (and, because you used `=` instead of a function, it throws a `ReferenceError` because `event` is unknown).

When you do:

```js
ws.onmessage = (event) => console.log(event.data);
```

you are **giving the WebSocket a piece of code (the arrow function) that it will call later**—exactly when a network packet arrives, the browser’s *event loop* pushes a task onto the queue and then invokes your callback with a proper `event` object.

### The mental model

| Step | What happens |
|------|--------------|
| **1️⃣ Assignment** | You hand the WebSocket a function reference. The connection is opened and the code continues. |
| **2️⃣ Waiting** | The WebSocket is listening for bytes on the network. The main thread is free to do other work (run timers, handle clicks, etc.). |
| **3️⃣ Data arrives** | The network layer builds an `event` object, places a *task* in the JavaScript *task queue*. |
| **4️⃣ Callback execution** | The event‑loop picks the task, calls your callback (`function(event){…}`), and you can safely read `event.data`. |

If you tried to “write the code directly” (e.g., a `while` loop waiting for a message) you would **block the single JavaScript thread** and the page would freeze—something browsers deliberately prevent.

---

## 3.  Under the hood: the event loop

1. **Synchronous code** runs to completion.  
2. When an asynchronous event (network packet, timer, user click) occurs, the browser enqueues a **task**.  
3. When the call‑stack is empty, the event loop dequeues the next task and runs its associated callback.

That’s why callbacks are the natural way to react to asynchronous events in JavaScript.

---

## 4.  More than one callback? Use `addEventListener`

You can attach several listeners to the same event:

```js
ws.addEventListener('message', (e) => console.log('A:', e.data));
ws.addEventListener('message', (e) => {
  // maybe update the UI
  document.getElementById('chat').textContent += e.data + '\n';
});
```

Setting `ws.onmessage` overwrites any previous handler; `addEventListener` lets multiple pieces of code react to the same incoming data.

---

## 5.  Benefits of the callback pattern

| Benefit | Why it matters |
|---------|----------------|
| **Non‑blocking** | The thread stays free to handle UI, timers, other WebSockets, etc. |
| **Decoupling** | The WebSocket library only knows *when* to call your code, not what it does. |
| **Composability** | You can pass different callbacks for different situations (e.g., separate handlers for “chat” vs. “system” messages). |
| **Closure** | Because the callback is a function, it can “see” variables from its lexical scope, letting you carry state easily. |

---

## 6.  Caveats & modern alternatives

- **Inversion of control** – you hand control of your code to the library. If the library never calls your callback (network failure, bug), your code may never run.  
- **Callback hell** – nesting many callbacks leads to deep indentation.  
- **Error handling** – if a callback throws, the error bubbles up to the event loop, which may swallow it or log it in a cryptic way.

Because of these, many newer APIs (including the `WebSocket` object in browsers) are being paired with **Promises** or **async/await** patterns, and libraries like `ws` (Node) expose a `message` event that works with the same callback idea but can be wrapped in a `Promise`:

```js
// Using a promise‑based wrapper (pseudo‑code)
function onceMessage(ws) {
  return new Promise(resolve => {
    ws.once('message', data => resolve(data));
  });
}

// later, with async/await:
const data = await onceMessage(ws);
```

But underneath the Promise is still a callback; it’s just sugar that makes the asynchronous flow easier to read.

---

## TL;DR

- **A callback is a function you give to something else so that thing can invoke it later, when a specific event (e.g., a WebSocket message) happens.**  
- **`ws.onmessage = handler`** registers *the reference* to that function. The JavaScript engine will call `handler` **when a message actually arrives**, not when the assignment line runs.  
- Writing the code directly would execute it immediately, before any data exists, and would block the single‑threaded runtime. Passing a function lets the WebSocket (or any other API) manage the timing and keeps your program responsive.

Feel free to experiment: open a WebSocket, attach a callback, and watch the console log every time the server pushes a payload. That’s the classic callback pattern in action!
