**Short answer:**  
`Uncaught SyntaxError: Unexpected token` means that JavaScript tried to evaluate a string that isn’t valid JavaScript (or JSON) syntax. When you see it “when a message arrives”, it almost always means you’re trying to **parse a message as JSON** (or run it through `eval`) and the raw payload isn’t valid JSON.

---

## 1. Why the error appears

| Where the error is thrown | Typical culprit | What the “unexpected token” actually is |
|---------------------------|----------------|------------------------------------------|
| `JSON.parse(str)`         | The server (or a routine) returns something that isn’t JSON. | A leading `<` (HTML page), a BOM (`\uFEFF`), a stray `\r`, an unquoted key, a trailing comma, `null`, `undefined`, etc. |
| `eval(str)` / `<script src="…">` injection | You (or a library) are trying to execute raw text as code. | Same as above – the parser sees a character it didn’t expect. |
| `Function("…")` or `setTimeout(str, …)` | Same as `eval`. | Same as above. |
| Implicit JSON‑parsing (e.g. `fetch(...).json()`) | The `Response` body isn’t a JSON document. | Same as above – the Fetch API tries to parse it for you. |

In short, **the “unexpected token” is the first character that the parser could not fit into the syntax it expected** (usually a JSON object or JavaScript expression).

---

## 2. Typical real‑world scenarios

| Scenario | What goes wrong | Example of the offending string |
|----------|-----------------|---------------------------------|
| **Server returns an HTML error page** (404, 500, etc.) instead of JSON. | The body starts with `<!DOCTYPE html>` or `<html>`. | `"<!DOCTYPE html><html>…"` |
| **Missing `Content‑Type: application/json`** → the client treats it as plain text, but you still call `JSON.parse`. | The server sends `"true"` (valid JSON) or `"abc"` (invalid). | `"true"` is OK, `"True"` (capital T) isn’t. |
| **Double‑encoding**: the server sends a *string* containing JSON, e.g. `"\"" + JSON.stringify(data) + "\""`. | After one parse you get a *string* that still looks like JSON, not an object. | `"{\"a\":1}"` → after `JSON.parse` you get a *string* `{"a":1}`, not an object. |
| **Trailing commas** in JSON (not allowed). | `"{\"a\":1,}"` | `,}` |
| **Invisible characters**: BOM, zero‑width spaces, etc. | The payload starts with `\uFEFF` (UTF‑8 BOM). | `"\uFEFF{"a":1}"` |
| **Missing quotes** around keys or values. | `"{a:1}"` | `a` (unexpected token `:`) |
| **Accidentally using `JSON.parse` on something that’s already an object**. | You pass an actual `Object`, not a string. | `JSON.parse({a:1})` → “unexpected token :” because the engine sees the colon of the object literal. |
| **WebSocket/SSE/Server‑sent messages that are not JSON**. | You blindly call `JSON.parse` on whatever the server pushes. | `"ping"` or `"Hello world"` |

## 3. How to pinpoint the exact cause

### 3.1. Log the raw message **before** you try to parse it

```js
socket.onmessage = event => {
    const raw = typeof event.data === 'string' ? event.data : event.data.toString(); // in case it’s a Blob/ArrayBuffer
    console.log('Raw incoming data:', raw);
    try {
        const data = JSON.parse(raw);
        // …use data…
    } catch (e) {
        console.error('JSON.parse failed:', e);
        // Keep the raw string around for inspection:
        console.table({ raw, error: e.message });
    }
};
```

If you already have a `fetch` call, use `.text()` first:

```js
fetch('/api/items')
  .then(r => r.text())
  .then(text => {
      console.log('Response body (raw):', text);
      // Now decide whether to parse:
      const data = JSON.parse(text);
  })
  .catch(err => console.error(err));
```

### 3.2. Open the **Network** tab

1. Filter by the request that delivers the message (XHR, Fetch, WebSocket, SSE).
2. Click the request → **Response** (or **Frames** for WebSocket).
3. Verify the body really is JSON. Look for:
   * HTML markup (`<`, `!DOCTYPE`, `html>`)
   * Leading/trailing whitespace or invisible characters
   * BOM (`EF BB BF` in hex)
   * Anything other than `{`, `[`, `"`, `true`, `false`, `null`, a number

### 3.3. Check the **Content‑Type** header

* For JSON‑only endpoints you should see `Content‑Type: application/json; charset=utf-8`.
* If it’s `text/html`, the server is returning an HTML page (often an error page).

### 3.4. Verify the payload is **well‑formed JSON**

Copy the raw text from the console/network and test it in a JSON validator (online or `JSON.parse` in a Node REPL). If it fails, you’ll see exactly where the parser chokes.

### 3.5. Look for **double‑encoding**

If you see something like `"{ \"a\": 1 }"` (quotes around the whole object) you’re dealing with a JSON string rather than an object. You may need to `JSON.parse` twice, or fix the server so it doesn’t wrap the JSON.

## 4. Common fixes (and when to apply them)

| Symptom | Fix | Example |
|---------|-----|---------|
| **HTML error page** (e.g., 404/500) | Fix the server to return proper JSON for error states, or handle non‑JSON responses before parsing. | ```js if (!response.ok) throw new Error('API error'); ``` |
| **Missing `Content-Type`** | Configure the server to set `Content-Type: application/json`. | In Express: `res.setHeader('Content-Type', 'application/json');` |
| **Trailing commas / stray characters** | Sanitize the JSON on the server, or `trim()` it client‑side. | `const cleaned = raw.trim().replace(/[\x00-\x1F\x7F]/g, '');` |
| **BOM or other invisible bytes** | Strip the BOM before parsing: `raw.replace(/^\uFEFF/, '')` or `raw.trim()`. | ```js const json = raw.replace(/^\uFEFF/, ''); const data = JSON.parse(json); ``` |
| **Double‑encoded JSON** | Parse twice only when you know the server deliberately sends a JSON string: `data = JSON.parse(JSON.parse(raw));`. Better yet, fix the server. | |
| **Trying to `JSON.parse` an already‑parsed object** | Pass the object directly: `const data = event.data; // already an object` | |
| **Using `eval` instead of `JSON.parse`** | Replace `eval(something)` with `JSON.parse(something)`. If you truly need to run dynamic code, use `new Function()` with strict mode and proper sandboxing. | |
| **WebSocket message that isn’t JSON** | First check the `type` field (if the protocol defines one) or simply decide: if it looks like JSON, parse; otherwise ignore. | ```js const isJson = raw.trimStart().startsWith('{') || raw.trimStart().startsWith('['); if (isJson) data = JSON.parse(raw); else console.log('Non‑JSON message:', raw); ``` |
| **Accidentally sending a comment** (e.g., `// comment`) in JSON | Remove comments from the JSON generator – JSON does not support comments. | |

## 5. A “safe‑parse” wrapper you can drop in

```js
/**
 * Try to parse `str` as JSON.
 * Returns { success: true, data } on success,
 * or { success: false, raw, error } on failure.
 */
function safeJsonParse(str) {
    if (typeof str !== 'string') {
        // If you already got an object, just return it.
        return { success: true, data: str };
    }
    try {
        // Strip BOM, trim whitespace and control chars.
        const cleaned = str.replace(/^[\uFEFF\x00-\x1F\x7F]+/, '').trim();
        return { success: true, data: JSON.parse(cleaned) };
    } catch (e) {
        return { success: false, raw: str, error: e };
    }
}

// Usage:
const result = safeJsonParse(event.data);
if (result.success) {
    // Use result.data
} else {
    console.error('Failed to parse message', result.error);
    // Maybe notify the user, retry, etc.
}
```

This pattern keeps the rest of your code blissfully unaware of malformed input, while still giving you a chance to log exactly what went wrong.

## 6. What to ask yourself (or us) when you’re stuck

1. **What technology delivers the message?**  
   - Fetch / XHR?  
   - WebSocket?  
   - Server‑Sent Events?  
   - Service‑Worker `postMessage`?

2. **What does the raw payload look like?** (Copy it from the console / Network tab.)

3. **What is the `Content‑Type` of the response?** (If it’s a HTTP request.)

4. **Are you doing anything else with that data besides JSON.parse?** (e.g., `eval`, inserting it into the DOM as HTML, etc.)

5. **Can you reproduce the problem consistently?** (e.g., “every time I click the ‘Submit’ button”.)

Providing the relevant snippet (the `onmessage` handler, the fetch call, etc.) together with a sample of the raw message will let us pinpoint the exact problem.

## 7. TL;DR Checklist

- [ ] **Log the raw message before you touch it** (console.log, Network tab).
- [ ] **Check it’s actually JSON** (starts with `{` or `[`, no leading `<`, no BOM, no stray whitespace).
- [ ] **Make sure the server sends `Content-Type: application/json`** (or handle non‑JSON explicitly).
- [ ] **Trim / strip BOM** before `JSON.parse` if you suspect invisible characters.
- [ ] **Never `eval` JSON**; use `JSON.parse`.
- [ ] **Wrap parse in try/catch** (or the `safeJsonParse` helper) so you can see *what* failed.
- [ ] **If you get HTML**, the server is returning an error page—fix the server or handle HTTP errors.
- [ ] **If you get double‑encoded JSON**, either `JSON.parse` twice or fix the server.
- [ ] **If the payload is already an object**, don’t call `JSON.parse` on it.

Follow those steps, and you’ll find the exact character that’s “unexpected” and be able to either clean the data or fix the sender. Feel free to share the snippet and the raw text if you need a more concrete suggestion!
