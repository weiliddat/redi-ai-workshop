That's an excellent question that cuts to the heart of **asynchronous programming** and **event-driven paradigms** in JavaScript (and many other languages). Let's break it down clearly, using your `ws.onmessage` example as the anchor.

### 1. What is a Callback Function? (The Simple Definition)
A **callback function** is a function you **pass as an argument** to another function, with the understanding that the receiver function will **call it back (execute it) at a later time**, typically when a specific event occurs or an asynchronous operation completes.

**Key Insight**: You don't call the callback yourself *right now*. You entrust it to another piece of code (like the WebSocket API, a timer, or a network request) to invoke it **when it's appropriate** (e.g., when data arrives, a timeout expires, or a button is clicked).

---

### 2. Why *Not* Write the Code Directly? (The Core Problem)
Let's imagine you tried to handle a WebSocket message **without** using a callback. You might mistakenly write something like this:

```javascript
// ❌ WRONG: This does NOT work for handling future WebSocket messages!
ws.onmessage = console.log("Received: " + event.data); // <-- Imagine event.data exists here
```

**Why this fails catastrophically**:
- **`console.log(...)` executes IMMEDIATELY** when this line runs (when you set up the WebSocket), **not** when a message arrives.
- At the exact moment you assign `ws.onmessage`, **no message has arrived yet** (the connection might not even be open!). So `event.data` is `undefined`, and you're logging nonsense *right now*.
- **Worse**: You've just overwritten `ws.onmessage` with the *return value* of `console.log` (which is `undefined`). Now, when a real message *does* arrive later, the WebSocket tries to call `undefined` as a function → **TypeError: undefined is not a function**.

---

### 3. How the Callback Pattern Solves This (The Right Way)
Here's the correct approach using a callback:

```javascript
// ✅ CORRECT: Pass a FUNCTION (the callback) to be called LATER
ws.onmessage = function(event) {
    console.log("Received: " + event.data);
};

// Or equivalently with an arrow function:
ws.onmessage = (event) => {
    console.log("Received: " + event.data);
};
```

**Why this works**:
- You're **not calling the function now**. You're **assigning the function definition itself** (a reference to the code) to `ws.onmessage`.
- The WebSocket API internally stores this function reference.
- **Later**, when a message actually arrives from the server, the WebSocket **looks up your stored function** and **calls it back** (invokes it), passing the `event` object as an argument.
- Your code inside the function **only runs at that exact future moment** when data is available.

---

### 4. Real-World Analogy: The Restaurant Pager
Think of it like this:
- You go to a busy restaurant and put your name on the list.
- Instead of standing there staring at the host stand (blocking your time), you **give the host a callback**: *"When my table is ready, please call me on this pager number."*
- The host **doesn't call you immediately**. They hold onto your pager number (the callback reference).
- **When your table is ready** (the asynchronous event occurs), the host **calls your pager** (invokes your callback).
- You only stop what you're doing and go to the table **when the pager goes off** (when your code runs).

**If you tried to "do it directly"**: You'd stand frozen at the host stand the whole time, waiting. You couldn't leave, check your phone, or do anything else—because you're blocking the thread waiting for the table. That's how synchronous code would work (and it's terrible for UX!).

---

### 5. Why This Pattern is Everywhere in JavaScript
JavaScript is single-threaded and non-blocking by design (especially in browsers). If you waited synchronously for every network request, timer, or user input, your entire page would freeze. **Callbacks (and their modern successors like Promises/async-await) are how we avoid blocking the main thread.**

Other everyday examples:
- `setTimeout(myCallback, 1000)` → `myCallback` runs ~1 second later.
- `button.addEventListener('click', myCallback)` → `myCallback` runs when the button is clicked.
- `fetch(url).then(myCallback)` → `myCallback` runs when the network request completes.

**In all cases**: You're passing a function to be executed **in the future**, not now.

---

### Key Takeaway for Your Question
> You pass a function to `ws.onmessage` (or any event handler) **because the code that handles the event cannot run at the time you set up the handler**. It must run **later**, when the asynchronous event (a WebSocket message) actually occurs. Writing the code directly would execute it immediately (too early, with wrong data) and break the event handler for future events.

This isn't just a quirk of WebSockets—it's the **foundation of responsive, non-blocking web applications**. Once you grasp that callbacks delegate future work to the browser/runtime, event-driven code makes perfect sense.
