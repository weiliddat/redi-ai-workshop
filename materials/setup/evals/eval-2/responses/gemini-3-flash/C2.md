To understand a **callback function**, it helps to think of it as a "to-do list" you give to someone else to run when they are ready.

### 1. What is a Callback Function?
A callback is a function passed as an **argument** to another function. The "receiving" function then "calls it back" (executes it) later, usually after a specific task is finished or an event occurs.

### 2. Why pass a function to `ws.onmessage` instead of just writing code?

If you wrote your code directly like this:
```javascript
// This won't work the way you expect
ws.onmessage = console.log("I got a message!"); 
```
The computer would run `console.log` **immediately** the moment it reads that line. It wouldn't wait for a message to arrive from the server; it would just print to the console once and then `onmessage` would be set to `undefined` (the result of the log).

**By passing a function, you are providing a "plan" for the future.**

Here are the three main reasons why we do this:

#### A. The "Wait and See" (Asynchrony)
WebSockets are **asynchronous**. You have no idea when a message will arrive—it could be in 10 milliseconds or 10 minutes. 
*   If you wrote code directly, your program would have to **stop and freeze** (block) until a message arrived. 
*   By using a callback, your code keeps running. You are essentially telling the browser: *"Go about your business; but IF the 'message' event happens, run this specific block of code for me."*

#### B. The "Payload" (Data Handling)
The callback function acts as a wrapper that receives data. When the WebSocket gets a message, it "calls" your function and **hands it the data** as an argument:
```javascript
ws.onmessage = function(event) {
    console.log(event.data); // The browser 'pushed' the data into your function
};
```
If you just wrote code directly, you would have no way to "catch" the incoming data from the server.

#### C. Reuse and Logic Separation
Passing a function allows you to define different reactions for different scenarios. You might want one WebSocket to update a chat box and another to update a stock ticker. By passing functions, you tell the same underlying system (`ws`) to behave differently based on the "recipe" (callback) you provide.

### A Real-World Analogy: The Pizza Order

Imagine you are ordering a pizza over the phone:

*   **Doing it "Directly" (Synchronous):** You call the pizza shop and stay on the phone. You don't hang up. You stand there silently for 20 minutes listening to the oven. You can't do anything else until the pizza is ready.
*   **The Callback (Asynchronous):** You call the shop, give them your number, and **hang up**. You go watch TV or clean your room. When the pizza is ready, they **call you back**. 

The **phone number** is the callback function. You aren't "running" the conversation yet; you are giving them the means to start a conversation with you once the "pizza is ready" event occurs.
