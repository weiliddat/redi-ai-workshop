The issue is not in the client-side code you provided. Your HTML/JavaScript logic correctly waits for the **Server** to send a message back to the browser before displaying it.

Since your code does **not** add the message to the screen immediately (when you click Send) and only adds it inside `socket.onmessage`, the fact that the message appears twice means **your server is sending it twice.**

Here is the breakdown of why this happens and how to fix it.

### 1. The Root Cause: Server Logic
In a typical WebSocket chat server (like Node.js/Express, Python, etc.), when you send a message, the server usually does one of two things:
1.  **Broadcasts** the message to all connected users (including you).
2.  **Explicitly sends** a confirmation or echo back to the sender.

If your server code loops through *all* connected clients to broadcast the message **AND** also sends the message specifically back to the sender, you (as the sender) will receive the message twice:
*   **Message #1:** You receive it because you are part of the "everyone" group.
*   **Message #2:** You receive it because the server sent it specifically to your socket ID.

### 2. The Fix (Server-Side)
You need to modify your server code. Most chat servers have a loop that looks something like this:

**Incorrect Server Logic:**
```javascript
// Server sends to ALL clients (including YOU)
clients.forEach(client => {
    client.send(message);
});

// ALSO SENDS to YOU explicitly (causing the double)
sender.send(message); 
```

**Correct Server Logic:**
Skip sending it to yourself when broadcasting to others.
```javascript
clients.forEach(client => {
    // If the client is NOT the one who sent the message, send it
    if (client !== sender) {
        client.send(message);
    }
});

// Note: You might want to manually send to the sender separately if you need them to see it, 
// but often the broadcast is enough as the sender is also receiving the message 
// as part of the room history (if your server architecture supports it).
```

### 3. Alternative Fix (Client-Side Workaround)
If you cannot change the server code (e.g., it's a third-party API), you can filter out your own second message on the client side.

1.  Store your name in a variable.
2.  Check if the incoming message name matches yours inside `socket.onmessage`.
3.  If it matches, **do not** render the message if you have already received it locally (or if the goal is to ensure no duplicates).

*Note: Since your current code does not render messages locally when clicking "Send", filtering self might make you not see any messages at all if the server intends to handle your own echo.*

*   [ ] **Does your server run a loop for *all* clients?** -> Ensure `if (client !== sender)` is used inside the loop.
*   [ ] **Did you edit the code?** Did you (or a tutorial you copied from) add an `addMessage` call inside the `chatForm.submit` handler? Your current code provided has it, so ensure you didn't accidentally uncomment or add that line earlier.
*   [ ] **Browser Cache:** Hard refresh the page (Ctrl+F5 or Cmd+Shift+R) to ensure you aren't running cached Javascript.

**Summary:** Your client code is fine. You need to check your Node.js/Python backend and ensure you aren't broadcasting the message to yourself **in addition** to the direct send, or you are looping over clients that include the sender, and also sending directly to the sender.
