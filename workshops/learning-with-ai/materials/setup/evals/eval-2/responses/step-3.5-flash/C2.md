## What is a Callback Function?

A **callback function** is a function passed as an argument to another function (or assigned to an event property) to be executed **later** when a specific event occurs or an operation completes. It's essentially "calling back" to code you provided.

In your WebSocket example, `ws.onmessage` is a **property** that expects a function. That function will be invoked automatically by the browser/WebSocket whenever a new message arrives.

```javascript
// This is a callback function assigned to ws.onmessage
function handleMessage(event) {
  console.log('Received:', event.data);
}

ws.onmessage = handleMessage;  // assigning the callback
```

---

## Why Not Write the Code "Directly"?

You actually **can** write it directly using an *anonymous function*, which is still a callback:

```javascript
// "Direct" version (anonymous callback)
ws.onmessage = function(event) {
  console.log('Received:', event.data);
  // process message...
  updateUI(event.data);
  saveToDatabase(event.data);
};
```

Both approaches are callbacks! The key insight is that **you must provide a function** - not the result of running code.

### Why Use Named vs. Anonymous Callbacks?

#### 1. **Reusability** (named functions)
```javascript
function handleMessage(event) {
  console.log('Received:', event.data);
}

// Reuse in multiple places
ws1.onmessage = handleMessage;
ws2.onmessage = handleMessage;
```

#### 2. **Readability & Debugging**
```javascript
// Anonymous (harder to debug)
ws.onmessage = function(event) { /* ... */ };

// Named (appears in stack traces)
function handleWebSocketMessage(event) { /* ... */ }
ws.onmessage = handleWebSocketMessage;
```

#### 3. **Separation of Concerns**
```javascript
// Setup WebSocket
const ws = new WebSocket('wss://example.com');

// Define behavior separately
ws.onmessage = handleIncomingMessage;
ws.onerror = handleError;
ws.onclose = handleConnectionClose;
// ... later in the code
function handleIncomingMessage(event) {
  // Dedicated function for message logic
}
```

#### 4. **Conditional Assignment**
```javascript
if (isDebugMode) {
  ws.onmessage = function(event) { 
    console.debug('Debug message:', event); 
  };
} else {
  ws.onmessage = handleProductionMessage;
}
```

## What Happens If You Don't Use a Function?

```javascript
// WRONG - This runs IMMEDIATELY, not when message arrives!
ws.onmessage = console.log('This prints now!');  // undefined assigned to onmessage

// WRONG - Tries to assign the result (undefined) instead of a function
ws.onmessage = someProcessingFunction(event);  // error: event not defined
```

The WebSocket **needs a function reference** to call later. When you write:
```javascript
ws.onmessage = function(event) { /* ... */ };
```
You're saying: "When a message arrives, **call this function** and pass the event object to it."

---

## Key Pattern: Event-Driven Architecture

This pattern appears everywhere in JavaScript:
```javascript
// Button click
button.addEventListener('click', function() { /* ... */ });

// Array iteration
array.forEach(function(item) { /* ... */ });

// HTTP request
fetch(url).then(function(response) { /* ... */ });
```

In all cases, you're providing a **function to be invoked later** by the system when the specific event/condition occurs.

---

## Modern Alternative: Arrow Functions

```javascript
// Concise anonymous callback with arrow function
ws.onmessage = (event) => {
  console.log('Received:', event.data);
};
```

---

## Summary

| What You Do | What Actually Happens |
|-------------|---------------------|
| `ws.onmessage = myFunction` | "Store this function reference" |
| `ws.onmessage = function()` | "Store this anonymous function" |
| **NOT** `ws.onmessage = myFunction()` | This **calls** `myFunction` immediately and assigns its return value (usually `undefined`) |

**The core principle**: You're not "writing code directly" vs. "using a callback" - **both approaches ARE callbacks**. The choice between named/anonymous functions is about code organization, reusability, and readability, not about whether it's a callback.

The essential rule: **`ws.onmessage` must receive a function reference** (callback), not the result of calling a function.
