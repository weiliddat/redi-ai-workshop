**`JSON.stringify` is required because WebSocket only sends strings or binary data (Blob/ArrayBuffer), not JavaScript objects directly.**

Here's why:

1. **WebSocket protocol limitation**: The WebSocket API's `send()` method accepts only:
   - Strings
   - Blob objects
   - ArrayBuffer/View objects
   - It does **not** accept plain JavaScript objects

2. **What happens if you don't stringify**:
   ```javascript
   // WRONG - sends "[object Object]"
   socket.send({ type: 'message', text: 'Hello' });
   
   // Server would receive the literal string: "[object Object]"
   // JSON.parse("[object Object]") would throw an error
   ```

3. **What `JSON.stringify` does**:
   ```javascript
   // CORRECT - sends a proper JSON string
   socket.send(JSON.stringify({ type: 'message', text: 'Hello' }));
   
   // Server receives: '{"type":"message","text":"Hello"}'
   // Can be parsed back to an object with JSON.parse()
   ```

4. **Analogy**: It's like mailing a physical object - you need to package it (serialize) first, then the receiver unpacks it (deserializes).

**Alternative approaches** (but JSON is standard):
- You could manually build strings: `socket.send('type=message&text=Hello')` (URL encoding)
- Use other serialization formats (MessagePack, protobuf, etc.)
- But JSON is: human-readable, built into JavaScript, and universally supported

**In your code specifically**:
- Client sends: `JSON.stringify({type:'message', text:'hi'})` → `'{"type":"message","text":"hi"}'`
- Server receives string → `JSON.parse()` → object
- Client receives string from server → `JSON.parse()` → object

Without stringification, your chat would simply send `[object Object]` every time, which the server couldn't parse.
