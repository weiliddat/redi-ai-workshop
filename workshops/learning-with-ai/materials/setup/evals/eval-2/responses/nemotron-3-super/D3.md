This error (`Uncaught SyntaxError: Unexpected token`) **almost always occurs when your JavaScript code tries to parse invalid JSON data** (typically via `JSON.parse()`) **when processing an incoming message** (e.g., from a WebSocket, Server-Sent Event, `fetch`/`XMLHttpRequest` response, or similar). Here's a breakdown of what's happening, why it occurs, and how to fix it:

### 🔍 **Root Cause**
Your code assumes incoming data is valid JSON, but the actual data received is **not valid JSON**. When `JSON.parse()` encounters unexpected characters (like `<`, `{`, `}`, plain text, HTML, or binary data), it throws this error.  
**Common examples triggering this:**
- Server accidentally returns an **HTML error page** (e.g., 404/500 error) instead of JSON → `JSON.parse()` sees `<` (start of `<html>`) → `Unexpected token '<'`.
- Server returns **plain text** (e.g., `"OK"` or an error message) → `JSON.parse()` sees `O` → `Unexpected token 'O'`.
- Malformed JSON (e.g., missing quotes, trailing commas, unescaped characters) → `JSON.parse()` fails.
- Binary data (e.g., image, protobuf) mistakenly treated as UTF-8 text → garbage characters break parsing.
- **WebSocket-specific**: Receiving a ping/pong frame or protocol error as a text message.

### 🛠️ **How to Diagnose & Fix**
Follow these steps:

1. **Log the raw incoming data *before* parsing**  
   In your message handler (e.g., `websocket.onmessage`, `fetch().then(res => res.text())`), **log the raw data**:
   ```javascript
   // BEFORE calling JSON.parse()
   console.log('Raw message:', event.data); // WebSocket
   // or
   const text = await response.text();
   console.log('Raw response:', text); // Fetch/XHR
   ```
   → **Check the DevTools Console** for the exact value causing the error.  
   → *Look for:*  
   - HTML tags (`<`, `>`, `</div>`)  
   - Plain text (`Error: Not found`, `OK`)  
   - Binary garble (`` or weird symbols)  
   - Empty string (`""`)  
   - `null` or `undefined` (if server sends `null` but you don't check)

2. **Verify the server's actual output**  
   - Open **DevTools → Network tab**.  
   - Find the request/WebSocket message causing the error.  
   - Inspect the **Response** (for HTTP) or **Frames** (for WebSocket).  
   - Confirm:  
     - Is the `Content-Type` header set to `application/json`? (Critical for HTTP!)  
     - Is the response body *actually* valid JSON? (Use [jsonlint.com](https://jsonlint.com) to validate).

3. **Handle non-JSON cases gracefully**  
   Never assume incoming data is valid JSON. Use a `try/catch`:
   ```javascript
   // WebSocket example
   socket.onmessage = (event) => {
     try {
       const data = JSON.parse(event.data);
       // Process valid JSON
       handleMessage(data);
     } catch (e) {
       console.error('Invalid JSON received:', event.data);
       // Handle error: log, reconnect, ignore, etc.
     }
   };

   // Fetch example
   fetch('/api/data')
     .then(res => {
       if (!res.ok) throw new Error(`HTTP ${res.status}`);
       return res.text(); // Get raw text FIRST
     })
     .then(text => {
       try {
         const data = JSON.parse(text);
         // Process data
       } catch (e) {
         console.error('Invalid JSON:', text);
         // Handle error
       }
     });
   ```

4. **Check for common server-side issues**  
   - **Authentication failure**: Server redirects to login page (returns HTML) instead of JSON.  
   - **Route misconfiguration**: Hitting a non-API endpoint (e.g., `/api/users` → serves HTML homepage).  
   - **Middleware error**: Express/Koa/etc. throws an unhandled error → sends HTML error page.  
   - **CORS/proxy interference**: A proxy or CDN modifies the response (e.g., injecting HTML ads/error pages).  
   - **Wrong endpoint**: Accidentally calling `/` instead of `/api/data`.

### 💡 **Prevention Tips**
- **Always validate `Content-Type`** for HTTP requests:  
  ```javascript
  if (response.headers.get('content-type') !== 'application/json') {
    throw new Error('Expected JSON response');
  }
  ```
- **Use schema validation** (e.g., [Zod](https://zod.dev/), [Joi](https://joi.dev/)) *after* parsing to catch structural issues early.
- **Enable detailed server logging** to see what the server *actually* sends when errors occur.
- **For WebSockets**: Implement heartbeat/ping-pong to detect broken connections early (prevents parsing garbage from dead connections).

### 🌰 **Real-World Example**
*Scenario:* Your WebSocket client expects `{ "type": "chat", "text": "Hi" }`, but the server returns an HTML login page because the user’s session expired.  
**Console error:**  
`Uncaught SyntaxError: Unexpected token '<' in JSON at position 0`  
**Fix:**  
1. Log `event.data` → see `<!DOCTYPE html>...`  
2. Check server logs → see `401 Unauthorized` triggering redirect to `/login.html`  
3. Solution: Handle auth errors on the server (return `{ "error": "unauthorized" }` as JSON) *or* handle redirects on the client (check for HTML before parsing).

### ⚠️ **Important Notes**
- The error **points to the `JSON.parse()` line**, but the *root cause* is the invalid data *before* it.  
- Do **not** ignore this error—it often indicates a deeper issue (auth failure, misconfigured routes, broken API).  
- If you see `Unexpected end of JSON input`, the server likely sent an **empty response** or **cut-off data** (e.g., network timeout).

By logging the raw data and validating assumptions, you’ll quickly identify whether the issue is client-side (parsing logic), server-side (incorrect response), or infrastructural (proxy/CDN). Start with step #1—it solves 90% of these cases! 🕵️‍♂️
