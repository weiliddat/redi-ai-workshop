## Adding `/nick` Command

You need to handle this in two places: **sending** the command and **receiving** the broadcast.

### 1. Detect the Command When Sending

In the `chatForm` submit handler, you currently always send a `message` type. Add a check for slash commands **before** that:

```javascript
chatForm.addEventListener('submit', (e) => {
  e.preventDefault();

  if (!socket || socket.readyState !== WebSocket.OPEN) {
    return;
  }

  const text = messageInput.value.trim();
  if (!text) return;

  // Add this block
  if (text.startsWith('/nick ')) {
    const newName = text.slice(6).trim();   // everything after "/nick "
    if (!newName) return;
    socket.send(JSON.stringify({
      type: 'nick',
      name: newName
    }));
    messageInput.value = '';
    return;                                 // stop here, don't send as a message
  }

  socket.send(JSON.stringify({
    type: 'message',
    text: text
  }));

  messageInput.value = '';
});
```

The `6` in `text.slice(6)` is the length of `"/nick "` (5 chars + 1 space).

---

### 2. Handle the Server's Response

The server broadcasts a `nick` type message to everyone. Add a branch in your `onmessage` handler alongside the existing `if/else if` chain:

```javascript
} else if (data.type === 'nick') {
  addMessage(`[${time}] ${data.oldName} is now known as ${data.name}`);
```

You also need to handle the **error case**, but that's already covered — you already handle `data.type === 'error'` which displays the "name already taken" message.

---

### What the flow looks like end to end

```
You type:    /nick AliceNew
Client sends: {"type": "nick", "name": "AliceNew"}

If name is free, server broadcasts to everyone:
  {"type": "nick", "oldName": "Alice", "name": "AliceNew", "timestamp": "..."}
  → everyone sees: "[14:30:00] Alice is now known as AliceNew"

If name is taken, server sends only to you:
  {"type": "error", "message": "name already taken"}
  → you see: "[ERROR] name already taken"
```

---

### One thing to be aware of

The client doesn't track the local name after a nick change. That's fine here because the **server owns the name** — the server uses whatever name it has on record when broadcasting your messages. You don't need to update `nameInput` or any local variable, though you could update `nameInput.value` for clarity if you wanted to.
