**Short answer:**  
The message  

```
WebSocket connection to 'ws://localhost:3000' failed
```  

means that the browser tried to open a WebSocket (WS) connection to `ws://localhost:3000`, but the handshake never completed. In other words, the server didn’t respond with the HTTP 101 “Switching Protocols” response that is required to turn a normal HTTP request into a WebSocket.

---

## Why this happens

| Typical cause | What it looks like in the console / network tab |
|---|---|
| **Server not running** | `net::ERR_CONNECTION_REFUSED` – nothing is listening on port 3000. |
| **Wrong port or path** | You requested `ws://localhost:3000` but the WS server is on a different port (e.g., 3001) or behind a specific path (e.g., `/socket.io`). |
| **No WebSocket support on the server** | The server is a plain HTTP endpoint (Express, Django, etc.) and never performs the WS upgrade. |
| **Mixed‑content restriction** | Your page is served over **https** but you’re trying to use an insecure **ws** URL – browsers block that. |
| **Firewall / proxy / VPN** | The request is blocked before reaching localhost, or a corporate proxy doesn’t support WS upgrade. |
| **Server closes the connection immediately** | The server accepts the upgrade but then immediately closes the socket (e.g., due to an authentication failure, missing `Origin` header, or a bug). |
| **Wrong protocol (ws vs. wss)** | You’re trying to connect to a secure endpoint (`wss://`) but typed `ws://`, or vice‑versa. |

## How to debug it

1. **Check that the server is really running**  
   ```bash
   # Windows
   netstat -ano | findstr :3000
   
   # macOS / Linux
   sudo lsof -i :3000
   ```
   If you see nothing, the server isn’t listening.

2. **Verify the URL your client is using**  
   ```js
   // Example (native WebSocket)
   const ws = new WebSocket('ws://localhost:3000/socket.io/?EIO=4&transport=websocket');
   ```
   *If you’re using Socket.IO, you usually don’t need to add the path manually – the library adds `/socket.io/` for you.*

3. **Look at the Chrome DevTools**  
   - Open the **Network** tab, filter by **WS** (or “WebSocket”), and reload the page.  
   - Click the request and inspect the **Headers** and **Frames** panes.  
   - If you see a **101 Switching Protocols** response, the handshake succeeded – the problem is later (e.g., the server immediately closes).  
   - If you see a **404** or **400** (or nothing), the server rejected the request.

4. **Test with an external tool**  
   ```bash
   # Using wscat (install with: npm i -g wscat)
   wscat -c ws://localhost:3000
   ```
   If this also fails, the server definitely isn’t providing WS on that URL.

5. **Check for mixed‑content**  
   - If your page is served via **https**, use **wss** (secure WebSocket) and a valid TLS certificate.  
   - If you’re on **http**, using **ws** is fine, but you can’t upgrade to **wss** without a certificate.

6. **Inspect server logs**  
   Most Node.js WS libraries log something when a client connects (or when they reject). Look for messages like `WebSocket connection from …` or `Upgrade required`.

7. **Firewall / corporate proxy**  
   If you’re behind a corporate VPN or a strict firewall, try disabling the VPN temporarily or running the test on a different network to see if that resolves the issue.

## Quick “what‑to‑fix” checklist

| Symptom | Quick fix |
|---|---|
| **Server not listening** | Start your WS server (or ensure it’s bound to the right port). |
| **Wrong path** | Add the correct path (e.g., `/socket.io` for Socket.IO) or mount the WS server on the same HTTP server as your app. |
| **Mixed‑content** | Change `ws://` to `wss://` **and** serve the page over HTTPS (or serve the page over HTTP). |
| **Same port conflict** | If you have an HTTP server (Express, etc.) on the same port, attach the WS server to that HTTP instance instead of creating a separate one. |
| **CORS / Origin** | If the server checks the `Origin` header, make sure you’re sending the expected origin or disable the check for development. |
| **Firewall** | Allow traffic on port 3000 or try a different port that isn’t blocked. |

## Example: A minimal Node.js WS server that works with the URL in the error

```js
// server.js
const http = require('http');
const WebSocket = require('ws');

const server = http.createServer((req, res) => {
  // You can keep a normal HTTP handler if you like
  res.writeHead(200, { 'Content-Type': 'text/plain' });
  res.end('OK\n');
});

// Create a WebSocket server attached to the same HTTP server
const wss = new WebSocket.Server({ server });

wss.on('connection', (ws) => {
  console.log('Client connected');
  ws.send('Welcome to the WebSocket server!');
});

server.listen(3000, () => {
  console.log('HTTP + WS server listening on http://localhost:3000');
});
```

Now the client can safely use:

```js
const ws = new WebSocket('ws://localhost:3000');
ws.onmessage = (event) => console.log('Message:', event.data);
```

and you should **no longer** see the “failed” error (provided the client code runs after the server is started).
