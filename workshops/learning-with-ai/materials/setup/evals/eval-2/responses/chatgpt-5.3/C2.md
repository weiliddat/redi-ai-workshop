A **callback function** is just a function you give to something else so it can call it later when something happens.

In your case, `ws.onmessage` is waiting for incoming messages from the server. You don’t control *when* a message arrives—it could be 1 second later or 10 minutes later. So instead of running code immediately, you pass a function that should run **when** the message eventually shows up.

Example:

```
ws.onmessage = function(event) {
  console.log(event.data);
};
```

You’re not running that function right away—you’re saying:  
“Hey, when a message arrives, call this.”

If you tried to “just write the code directly,” like:

```
console.log(event.data);
```

there’s a problem:
- There is no `event` yet
- No message has arrived yet
- There’s nothing to react to

So the callback is necessary because WebSockets are **event-driven** and **asynchronous**. The system needs a function it can trigger later, not code that runs immediately.

Quick analogy:  
It’s like ordering food and giving your phone number. The callback is the number—you’re saying “call me when it’s ready,” instead of standing at the counter forever.
